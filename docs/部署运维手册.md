# 「邻里圈」部署与运维手册

> 本文档涵盖 Docker 一键部署、生产环境配置、运维监控、备份恢复、成本分析等运维全流程。

## 一、30 分钟一键部署

### 1.1 前置条件

| 环境 | 要求 |
|------|------|
| 操作系统 | Linux (Ubuntu 20.04+ / CentOS 8+) / macOS / Windows WSL2 |
| Docker | 20.10+ |
| Docker Compose | 2.0+ |
| 内存 | ≥ 2GB |
| 磁盘 | ≥ 10GB 可用空间 |
| 端口 | 80（前端）、3000（API）、5432（DB）、6379（Redis） |

### 1.2 部署步骤

```bash
# 1. 克隆代码
git clone <repository-url> linli-circle
cd linli-circle

# 2. 配置环境变量
cp .env.example .env
# 编辑 .env，至少配置以下项：
#   JWT_SECRET           （必填，建议 32 字节随机字符串）
#   DB_PASSWORD          （必填，数据库密码）
#   REDIS_PASSWORD       （生产必填）
#   PII_ENCRYPT_KEY      （生产必填，64 个 hex 字符）
#   CORS_ORIGIN          （生产必填，访问域名，不能是 localhost）

# 快速生成密钥：
node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"

# 3. 一键启动
docker compose up -d

# 4. 验证部署
curl http://localhost:3000/api        # 返回 endpoints 列表
curl http://localhost:3000/api/health # 返回 { status: 'ok' }
open http://localhost                  # 打开前端
```

### 1.3 常用命令

```bash
# 查看服务状态
docker compose ps

# 查看实时日志
docker compose logs -f server         # 后端日志
docker compose logs -f client         # 前端日志
docker compose logs -f postgres       # 数据库日志

# 重启服务
docker compose restart server

# 停止服务
docker compose down                   # 停止并保留数据
docker compose down -v                # 停止并删除数据卷（慎用）

# 进入容器调试
docker compose exec server sh
docker compose exec postgres psql -U postgres -d linli_circle

# 数据库迁移（如需手动执行）
docker compose exec server npm run db:migrate
```

## 二、生产环境强化配置

### 2.1 必须配置的变量

| 变量 | 说明 | 生成方式 |
|------|------|---------|
| `JWT_SECRET` | JWT 签名密钥 | `node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"` |
| `DB_PASSWORD` | 数据库密码 | 强密码（≥12 字符，含大小写数字特殊字符） |
| `REDIS_PASSWORD` | Redis 密码 | 强密码 |
| `PII_ENCRYPT_KEY` | PII 加密密钥 | 32 字节 hex（64 个字符） |
| `CORS_ORIGIN` | 跨域来源 | 实际访问域名，如 `https://linli.example.com` |
| `NODE_ENV` | 运行环境 | 必须设为 `production` |

> ⚠️ **生产环境强校验**：若 `NODE_ENV=production` 但上述任一必填项缺失，服务启动时会自动退出。

### 2.2 数据库连接池调优

修改 `server/src/config/database.ts`（或通过环境变量）：

```typescript
pool: new Pool({
  max: parseInt(process.env.DB_POOL_MAX || '20'),  // 单实例最大连接数
  idleTimeoutMillis: 30000,
  connectionTimeoutMillis: 2000,
  statement_timeout: 30000,                          // 单语句超时 30s
  idle_in_transaction_session_timeout: 60000,        // 事务空闲超时 60s
})
```

**多实例部署注意**：N 个实例 × `DB_POOL_MAX` ≤ PostgreSQL `max_connections`（默认 100），建议留 20% 余量。

### 2.3 Nginx 反向代理（推荐）

```nginx
# /etc/nginx/sites-available/linli
server {
    listen 80;
    server_name linli.example.com;
    return 301 https://$server_name$request_uri;
}

server {
    listen 443 ssl http2;
    server_name linli.example.com;

    ssl_certificate     /etc/letsencrypt/live/linli.example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/linli.example.com/privkey.pem;

    # 前端
    location / {
        proxy_pass http://localhost:80;
        proxy_set_header Host $host;
    }

    # API
    location /api/ {
        proxy_pass http://localhost:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # WebSocket 支持
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
    }

    # 静态资源缓存
    location ~* \.(js|css|png|jpg|jpeg|gif|ico|svg)$ {
        expires 7d;
        add_header Cache-Control "public, immutable";
    }
}
```

启用配置：
```bash
sudo ln -s /etc/nginx/sites-available/linli /etc/nginx/sites-enabled/
sudo nginx -t
sudo systemctl reload nginx
```

## 三、运维监控

### 3.1 健康检查

| 端点 | 用途 | 状态码 |
|------|------|--------|
| `GET /health` | 基础存活检查 | 200 = 健康 |
| `GET /health/metrics` | 系统指标 | 200 = 返回指标 |
| `GET /api` | API 状态 | 200 = 正常 |

```bash
# 添加到监控系统的探针示例
curl -fsS http://localhost:3000/health || alert "后端异常"
```

### 3.2 关键指标

通过 `/health/metrics` 获取：

| 指标 | 正常值 | 告警阈值 |
|------|--------|----------|
| `db.pool.totalCount` | < 80% max | ≥ 90% max |
| `db.pool.waitingCount` | 0 | > 5 |
| `redis.connected` | true | false |
| `process.uptime` | 持续增长 | < 60s（异常重启） |
| `process.memory.rss` | < 500MB | ≥ 1GB |
| `cron.lastRunAt` | < 5 分钟前 | ≥ 30 分钟前 |

### 3.3 日志查看

#### 实时日志
```bash
# 所有服务
docker compose logs -f

# 单个服务
docker compose logs -f server

# 最近 100 行
docker compose logs --tail=100 server
```

#### 结构化日志（pino）
后端使用 pino 输出 JSON 格式日志，便于采集到 ELK / Loki / Datadog：

```bash
# 查看错误日志
docker compose logs server | grep '"level":50'

# 查看特定用户操作
docker compose logs server | grep '"userId":"abc123"'
```

#### 审计日志
所有敏感操作（登录、下单、转账、注销）记录到 `audit_logs` 表：

```sql
-- 查询某用户最近操作
SELECT created_at, action, resource, ip, status
FROM audit_logs
WHERE user_id = 'xxx'
ORDER BY created_at DESC
LIMIT 50;
```

### 3.4 告警配置示例（Prometheus + Alertmanager）

```yaml
# prometheus-alerts.yml
groups:
  - name: linli-circle
    rules:
      - alert: ServerDown
        expr: up{job="linli-server"} == 0
        for: 1m
        annotations:
          summary: "邻里圈后端服务不可用"

      - alert: HighErrorRate
        expr: rate(http_requests_total{status=~"5.."}[5m]) > 0.1
        for: 5m
        annotations:
          summary: "5xx 错误率超过 10%"

      - alert: DBPoolExhausted
        expr: db_pool_used / db_pool_max > 0.9
        for: 5m
        annotations:
          summary: "数据库连接池使用率超过 90%"
```

## 四、备份与恢复

### 4.1 自动备份

项目已内置 `backup.service.ts`，定时任务每日凌晨 2:00 执行全量备份：

- 备份目录：环境变量 `BACKUP_DIR` 配置（默认 `backups/`）
- 文件命名：`backup_YYYYMMDD_HHMMSS.sql.gz`
- 保留策略：7 天自动清理

```bash
# 查看备份列表
ls -lh backups/

# 手动触发备份
docker compose exec server npm run backup

# 清理过期备份（自动，无需手动）
```

### 4.2 手动备份

```bash
# 导出数据库
docker compose exec -T postgres pg_dump -U postgres linli_circle | gzip > backup_$(date +%Y%m%d).sql.gz

# 备份上传目录（如有）
docker compose exec server tar -czf /tmp/uploads.tar.gz /app/uploads
```

### 4.3 数据恢复

```bash
# 1. 停止应用服务（避免写入冲突）
docker compose stop server client

# 2. 恢复数据库
gunzip -c backup_20260101_020000.sql.gz | docker compose exec -T postgres psql -U postgres linli_circle

# 3. 重启服务
docker compose start server client

# 4. 验证
curl http://localhost:3000/api/health
```

### 4.4 灾备建议

| 备份频率 | 保留时长 | 存储位置 |
|---------|---------|---------|
| 每日全量 | 7 天 | 本地 + OSS/S3 |
| 每周全量 | 4 周 | OSS/S3 |
| 每月全量 | 12 月 | 异地冷存储 |

RPO（数据丢失容忍）：< 24 小时
RTO（恢复时间目标）：< 1 小时

## 五、成本效益分析

### 5.1 服务器配置建议

| 用户规模 | CPU | 内存 | 磁盘 | 月度费用（云厂商参考） |
|---------|-----|------|------|---------------------|
| 1-1000 | 2 核 | 4GB | 40GB SSD | ¥100-200 |
| 1000-1万 | 4 核 | 8GB | 100GB SSD | ¥400-800 |
| 1万-10万 | 8 核 × 2 | 16GB | 500GB SSD | ¥1500-3000 |
| 10万+ | 16 核 × 4 | 32GB | 1TB SSD + 读写分离 | ¥5000+ |

### 5.2 AI 大模型成本

- 假设单次 LLM 调用 500 tokens，单价 ¥0.002/1k tokens（GPT-3.5 级别）
- 每次匹配调用 1 次 → 单次成本约 ¥0.001
- 每日 1000 次推荐 → 每日成本约 ¥1
- 每月 AI 总成本：约 ¥30-300（取决于用量）

**降本策略**：
- 未配置 API Key 时自动降级为规则匹配（零成本）
- 使用国内模型（通义千问/智谱）成本可降至 1/3
- 增加本地缓存减少重复调用

## 六、扩展方案

### 6.1 水平扩展

```bash
# 启动多个后端实例
docker compose up -d --scale server=3

# 配合 Nginx upstream
upstream linli_servers {
    server localhost:3000;
    server localhost:3001;
    server localhost:3002;
}
```

**注意事项**：
- Token 黑名单、幂等缓存、WebSocket 必须使用 Redis（项目已支持）
- 定时任务需分布式锁（仅单实例执行）

### 6.2 数据库扩展

| 阶段 | 方案 |
|------|------|
| 1 万用户 | 单机 PostgreSQL + Redis |
| 10 万用户 | 主从复制 + PgBouncer 连接池 |
| 100 万用户 | 读写分离 + 分库分表 + 缓存前置 |

### 6.3 监控扩展

| 阶段 | 推荐方案 |
|------|---------|
| 初期 | Docker logs + curl 探针 |
| 成长 | Prometheus + Grafana + Alertmanager |
| 规模化 | ELK / Loki + APM（SkyWalking / Datadog） |

## 七、常见问题排查

### Q1: 启动报错 "JWT_SECRET 必须在 .env 中配置"
A: `.env` 文件中 JWT_SECRET 缺失或为默认值。

### Q2: 数据库连接失败
A: 检查 `DB_HOST`、`DB_PORT`、`DB_PASSWORD`，确认 PostgreSQL 容器已启动：
```bash
docker compose ps postgres
docker compose logs postgres | tail -20
```

### Q3: 前端 404
A: 确认 `CORS_ORIGIN` 配置正确（不能是 localhost），且 Nginx 反代配置正确。

### Q4: WebSocket 频繁断开
A: 检查 Nginx 配置是否支持 WebSocket 升级（参考 §2.3）。

### Q5: AI 推荐为空
A: 未配置 `AI_API_KEY` 时自动降级为规则匹配，属正常现象。如需启用，检查 Key 是否有效。

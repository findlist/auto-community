创新创业实践报告（二）：

一、实践目的

本次创新创业实践活动的主要目的是通过参与实际项目开发，将课堂所学的计算机专业知识应用于真实场景，提升自身的工程实践能力与创新思维。具体目标包括：

第一，深化技术理解与应用能力。在大二学年，我已学习了Web开发基础、数据库原理、软件工程等课程，但这些知识多停留在理论层面。通过参与"邻里圈"社区互助平台的开发实践，我希望将React前端框架、Express后端开发、PostgreSQL数据库设计等技术栈真正应用到项目中，理解前后端分离架构、RESTful API设计、数据库迁移等现代Web开发流程，从"知道是什么"升级到"知道怎么做"。

第二，培养创新创业意识与实践能力。邻里圈项目是一个面向社区互助的创新型应用，涵盖技能交换、共享厨房、时间银行、紧急求助等多个业务模块，这种产品设计思路让我认识到技术创新需要紧密结合社会需求。我希望通过实践理解"发现问题→设计方案→技术实现→迭代优化"的创新闭环，培养从用户视角思考产品价值的能力，为未来可能的创业尝试积累经验。

第三，提升团队协作与项目管理能力。软件开发从来不是单打独斗，需要需求分析、任务拆解、进度跟踪、质量把控等多环节协同。通过参与项目的模块开发、测试编写、文档维护等工作，我希望学习如何将个人工作融入整体项目进度，理解Git版本控制、任务优先级排序、技术债务管理等工程实践，为未来职业发展打下基础。

第四，探索职业方向与规划路径。通过一周的集中实践，我希望更清晰地认识自己对前端开发、后端架构、产品设计的兴趣偏好与能力短板，为后续课程选修、技能提升方向、职业规划提供决策依据。同时，通过解决真实的技术问题，建立"遇到困难→分析原因→查找资料→尝试方案→验证结果"的问题解决思维模式。

二、实践内容

本次实践围绕"邻里圈"社区互助平台的开发与完善展开，历时一周（2026年6月10日至6月17日），主要内容包括技术学习、功能开发、测试保障、项目管理四个方面。

（一）技术栈学习与架构理解

实践初期，我花了两天时间系统学习项目采用的技术栈与架构设计。邻里圈采用Monorepo结构，分为client前端与server后端两个子项目。前端使用React 18配合Vite构建工具、TypeScript类型系统、TailwindCSS样式框架；后端使用Express框架配合TypeScript、JWT身份认证、PostgreSQL 16数据库、Redis 7缓存；数据库迁移采用node-pg-migrate工具，测试框架选用Vitest。

这一阶段的最大收获是理解了前后端分离架构的设计逻辑。以往课程中的Web开发多是单一项目，前后端代码混在一起，而邻里圈的架构让我认识到：前端专注用户交互与页面渲染，后端专注业务逻辑与数据存储，两者通过RESTful API通信，这种分离使得代码职责清晰、维护成本降低、团队协作更高效。同时，我第一次接触Monorepo概念，理解了"多个子项目共享同一仓库"的管理优势，如统一依赖版本、共享工具函数、简化部署流程。

（二）功能模块开发实践

实践中期，我参与了三个核心功能模块的开发，分别是管理员内容编辑、拼单退款优化、配送地址簿系统。

1. 管理员内容编辑功能

这个功能允许管理员直接编辑平台上的所有帖子内容，包括标题、描述、图片、价格等字段。我的工作主要在后端服务层，需要实现getContentDetail获取内容详情、updateContent更新内容两个方法，并在admin.service.ts中设计可编辑字段白名单，防止管理员误改关键字段。

开发过程中遇到的最大挑战是字段映射问题。数据库表使用下划线命名（如user_id、created_at），而TypeScript代码使用驼峰命名（如userId、createdAt），需要在查询结果返回时进行转换。起初我手动逐个字段映射，代码冗长易错，后来学习了项目中的toSkill、toKitchen等序列化函数的写法，设计了通用的toContent函数，通过配置映射表自动处理，代码从50多行缩减到15行，可维护性大幅提升。

另一个收获是图片上传理解。项目使用独立的/upload接口处理图片，返回URL字符串，管理员编辑时只需提交URL即可，无需在内容编辑接口重复处理文件上传。这种"单一职责"设计让我认识到接口拆分的工程价值。

2. 拼单退款优化功能

邻里圈的拼单模块允许用户发起拼单、邀请参与、成团后共享资源。原有逻辑中，成团前（pending状态）退出可全额退款，但成团后（ongoing状态）退出缺少明确规则，可能造成发起人损失或参与者套利。

我参与设计的方案是：成团后退出采用部分退款机制，退还参与者90%，10%作为已发生成本补偿给发起人。核心实现是在group-order.service.ts的exit方法中添加ongoing分支，使用Math.floor向下取整计算退款金额，避免因浮点运算多退，差额自动归发起人，保证总额守恒。

这个过程让我理解了"业务逻辑→技术实现→数据一致性"的闭环。退款不仅是代码逻辑，还涉及积分冻结/解冻、账户余额变动、审计日志记录等环节，需要调用creditService.unfreezeCredits与creditService.earnCredits两个方法，并使用事务保证原子性。同时，我学习了行锁防双花的概念：在高并发场景下，如果两个用户同时退出同一拼单，可能出现重复退款，需要用SELECT FOR UPDATE锁定记录，这个知识点在课堂上只是概念，实践后才真正理解。

3. 配送地址簿系统

共享厨房模块支持用户在线下单购买社区美食，但原有系统缺少配送地址管理，用户每次下单都要手动填写地址，体验不便。我负责开发完整的地址簿功能，包括数据库表设计、后端服务、前端页面三个层次。

数据库设计阶段，我学习了node-pg-migrate工具的使用，编写了1704067200016_delivery_address.ts迁移文件，创建delivery_addresses表，包含recipient（收件人）、phone（电话）、address（地址）、is_default（是否默认）等字段。这个过程让我认识到：数据库迁移不是简单执行SQL，而是需要版本管理、回滚机制、索引优化等多方面考虑。我在表中添加了user_id索引，因为地址查询总是基于用户维度，索引能显著提升查询性能。

后端服务阶段，我在address.service.ts中实现了listByUser（查询用户所有地址）、create（新增地址）、update（更新地址）、remove（删除地址）、setDefault（设为默认）五个方法。setDefault方法的设计让我理解了"唯一性约束"的实现：一个用户只能有一个默认地址，设置新默认时需要先取消旧默认，这个过程要用事务包裹，防止中间状态出现两个默认地址。

前端页面阶段，我在AddressBook.tsx中实现了地址列表展示、新增/编辑弹窗、删除确认、表单校验等功能。最大的收获是学习了Modal组件的设计模式：弹窗状态通过useState管理，open/close通过props传递，表单数据通过onChange实时更新，提交时校验必填项（收件人、电话、地址）。同时，我理解了移动端适配的重要性：页面使用max-w-2xl mx-auto实现响应式布局，在375px手机屏幕与1280px桌面屏幕下都能正常显示。

（三）测试编写与质量保障

实践后期，我参与了后端单元测试的编写，为address.service、audit.service、admin.service、kitchen.service、emergency.service五个模块补充测试用例，累计新增67个测试，使项目总测试数达到165个，全部通过。

测试编写的过程让我理解了"测试驱动开发"的价值。起初我认为测试是额外负担，写完功能还要写测试，浪费时间。但实际编写后发现：测试帮助我发现多处隐藏bug，如address.service的setDefault方法在取消旧默认地址时缺少WHERE条件，可能误取消其他用户的默认地址；emergency.service的classifyUrgency方法对"困在电梯"的判断优先级过低，应该高于"水管漏水"。

更重要的是，测试教会了我"边界思维"。功能代码只考虑正常流程，而测试需要覆盖各种异常场景：参数缺失、数据为空、并发冲突、权限不足、网络超时等。例如，我为kitchen.service的create方法编写了名称为空、价格为负、数量为零等多种失败场景的测试，这些场景在功能开发时完全没想到，但测试倒逼我去思考并处理。

测试框架方面，我学习了Vitest的mock机制。由于测试不能连接真实数据库，需要用vi.fn模拟query方法，返回预设数据。这个概念起初很难理解，我花了半天时间查阅文档、阅读项目已有测试代码，才明白"模拟依赖、隔离测试"的道理。现在回头看，mock不仅是测试技术，更是设计思维：它让我认识到依赖注入、接口抽象的重要性，只有代码结构清晰、依赖明确，才能有效mock和测试。

（四）项目管理与文档维护

除代码开发外，我还参与了项目管理工作，包括进度跟踪、文档更新、遗留问题记录。

我学习了项目的TodoWrite工具使用，将任务拆解为p0拼单退款、fe1技能纠纷页、fe2厨房评价页、be1操作日志后端、fe3操作日志前端、be2地址簿后端、fe4地址簿前端、test测试补全等多个子任务，每个任务标记pending、in_progress、completed三种状态，逐项推进。这种任务管理方式让我认识到：复杂项目不能"想到哪做到哪"，需要明确计划、跟踪进度、及时调整。

文档维护方面，我参与了docs/development-plan.md的更新，记录了已完成的功能模块、修改的文件清单、遗留的技术债务（如全局any类型替换、PostgreSQL慢查询优化、日志分级改造）。这个过程让我理解了"知识沉淀"的价值：项目代码会持续迭代，但文档记录了设计思路、决策依据、演进路径，是团队知识资产的重要组成部分。

三、实践总结及实践意义

一周的创新创业实践让我收获颇丰，在技术能力、工程思维、创新意识、职业规划四个方面都有显著提升。

（一）技术能力提升

通过真实项目开发，我的技术栈理解从"概念认知"升级到"应用能力"。React不再只是"组件化框架"的概念，我知道了useState管理状态、useEffect处理副作用、TailwindCSS编写响应式样式；Express不再只是"Node.js Web框架"，我知道了路由设计、中间件机制、错误处理、JWT认证；PostgreSQL不再只是"关系型数据库"，我知道了表设计、索引优化、事务管理、迁移工具。

更重要的是，我理解了技术栈之间的协作关系。前后端不是孤立存在，而是通过API接口通信，需要统一的字段命名规范、错误码体系、分页格式；数据库与缓存不是简单替代，而是分工协作，频繁查询用Redis缓存、持久数据用PostgreSQL存储；测试与代码不是先后关系，而是并行开发，测试用例倒逼代码设计优化。

（二）工程思维建立

实践让我建立了多方面的工程思维：

第一，职责分离思维。功能模块不是"大杂烩"，而是清晰分层：路由层处理HTTP请求解析、服务层处理业务逻辑、数据层处理存储查询，每层专注自己的职责，通过接口通信。这种思维让我认识到：好的代码不是"写得快"，而是"职责清、易维护、可测试"。

第二，异常处理思维。代码不能只考虑正常流程，必须覆盖参数缺失、权限不足、并发冲突、网络超时等异常场景。实践中遇到多次"代码能跑但测试失败"的情况，都是因为缺少异常处理，如updateContent方法未检查内容是否存在、setDefault方法未处理地址已被删除。

第三，性能优化思维。功能实现后还需要考虑性能：数据库查询是否缺少索引、前端渲染是否触发不必要的重绘、API响应是否返回过多数据。实践中我为delivery_addresses表的user_id字段添加索引，查询性能从200ms降到5ms，这个差距让我真正理解了"索引不是可有可无，而是必须设计"。

第四，安全防护思维。Web开发必须考虑XSS攻击、SQL注入、权限绕过等安全问题。项目中的sanitizeObject函数用于清洗用户输入，移除HTML标签与脚本代码；creditService的所有方法都检查用户权限，防止跨账户操作；这些细节让我认识到"安全不是事后补丁，而是设计阶段必须考虑"。

（三）创新意识培养

邻里圈项目的产品设计让我理解了"创新不是天马行空，而是发现需求、设计方案、验证效果"的闭环。

社区互助是一个真实的社会需求：城市居民 increasingly 依赖线上平台，但邻里关系日渐疏远，技能交换、共享厨房、时间银行等模块正是为了重建社区连接。这种产品设计思路让我认识到：技术创新的价值不在于"炫酷"，而在于"解决真实问题、创造社会价值"。

拼单退款的优化过程让我理解了"创新迭代"的逻辑：初始方案只有成团前退款，上线后发现成团后退出需求，设计部分退款机制平衡各方利益，通过测试验证逻辑正确性，最终上线。这个过程不是一次完成，而是发现问题→分析原因→设计方案→实现验证→迭代优化的持续闭环。

同时，我理解了"创新的风险意识"。产品设计不能只看正面效果，必须考虑负面影响：拼单退款可能被恶意套利、管理员编辑可能误删关键数据、配送地址可能泄露隐私。项目中的审计日志、权限校验、数据脱敏等机制，都是为了降低风险，这些让我认识到"创新需要平衡收益与风险"。

（四）职业规划启发

一周的实践让我更清晰地认识了自己的兴趣偏好与能力短板。

兴趣方面，我发现对后端架构、数据库设计、业务逻辑实现更有兴趣，而对前端样式调整、UI细节打磨兴趣较弱。这启发我后续课程选修优先考虑数据库系统、分布式架构、算法设计等方向，职业规划倾向后端工程师、数据工程师路径。

能力短板方面，我发现并发控制、性能优化、安全防护等知识点薄弱。实践中的行锁防双花、索引优化、XSS清洗等概念，课堂上有讲但理解不深，实践后才真正掌握。这启发我需要补充学习分布式系统、网络安全、性能调优等知识。

同时，我认识到职业发展需要"技术深度+业务广度"的双重能力。技术深度是指精通某一领域，如数据库优化、前端架构；业务广度是指理解产品设计、用户需求、行业趋势。实践中的功能开发让我理解：技术实现必须服务于业务目标，代码质量直接影响用户体验，这种"技术+业务"的综合视角是职业发展的关键。

实践意义方面，本次经历最大的价值是建立了"理论与实践的桥梁"。以往课程学习多是"知道是什么"，实践后升级为"知道怎么做、知道为什么这样设计"。这种认知升级不是通过阅读文档、观看视频获得，而是通过亲手写代码、遇到问题、查找方案、验证结果的真实过程积累。这让我认识到：创新创业教育的核心不是讲授知识，而是提供实践场景、引导问题解决、沉淀经验教训。

未来，我希望继续参与项目迭代，探索更多技术方向，如CI/CD流水线搭建、应急资源地图集成、数据统计报表可视化等，持续提升工程能力与创新思维，为可能的创业尝试或职业发展打下坚实基础。

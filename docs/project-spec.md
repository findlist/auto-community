# 「邻里圈」社区互助生态平台 - 项目规格说明

## 1. 项目背景

### 1.1 社会背景
随着城市化进程加快，邻里关系逐渐疏远。社区居民面临着：
- **孤独感加剧**：独居青年、空巢老人缺乏社交
- **资源浪费**：每家每户重复购买工具、设备
- **互助需求**：紧急时刻需要邻居帮助，但缺乏连接渠道
- **技能闲置**：很多人有技能但没有分享的平台

### 1.2 赛事背景
本项目参加 **TRAE AI 创造力大赛**，赛道为 **社会服务/造种新体验**。
- 赛事主题："世界很大，放手去造"
- 核心理念：用 AI 技术解决真实社会问题
- 参赛工具：TRAE IDE + TRAE Work

### 1.3 项目定位
**"一个平台，四种连接"** —— 打造有温度的社区互助生态

| 维度 | 描述 |
|------|------|
| 目标用户 | 社区居民（青年、老人、家庭） |
| 核心价值 | 重建邻里连接，实现资源互助 |
| 差异化 | 不是冷冰冰的互助平台，而是有烟火气的社区 |

## 3. 统一规范

### 3.1 API 响应格式

所有 API 接口统一返回以下格式：

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

**错误响应**：
```json
{
  "code": 400,
  "message": "参数错误",
  "errors": [
    {
      "field": "phone",
      "message": "手机号格式不正确"
    }
  ]
}
```

**错误码规范**：
| 错误码 | 说明 |
|--------|------|
| 200 | 成功 |
| 400 | 请求参数错误 |
| 401 | 未授权 |
| 403 | 禁止访问 |
| 404 | 资源不存在 |
| 429 | 请求过于频繁 |
| 500 | 服务器内部错误 |

### 3.2 分页响应格式

```json
{
  "code": 200,
  "data": {
    "total": 100,
    "page": 1,
    "page_size": 20,
    "list": []
  }
}
```

### 3.3 认证规范

- 使用 JWT Token 进行身份认证
- Token 有效期：7天
- Refresh Token 有效期：30天
- 请求头格式：`Authorization: Bearer <token>`

### 3.4 数据库设计规范

- 所有表必须包含：`id`, `created_at`, `updated_at` 字段
- 使用 UUID 作为主键
- 时间字段使用 TIMESTAMP 类型
- 地理位置使用 POINT 类型
- 软删除使用 `deleted_at` 字段

## 5. 数据模型设计

### 5.1 核心实体关系

```
User (用户)
  ├── UserProfile (用户资料)
  ├── UserSkills (用户技能)
  ├── UserCredits (用户积分)
  ├── UserReputation (用户信誉)
  └── UserLocation (用户位置)

HelpRequest (互助请求)
  ├── RequestCategory (请求类别)
  ├── RequestLocation (请求位置)
  ├── RequestStatus (请求状态)
  └── RequestResponse (响应记录)

Transaction (交易记录)
  ├── TransactionType (交易类型：积分/时间)
  ├── Sender (发送方)
  └── Receiver (接收方)
```

## 6. AI 能力设计

### 6.1 智能匹配算法

```
输入：用户A的技能/需求 + 用户B的技能/需求
处理：
  1. 技能相似度计算（NLP语义匹配）
  2. 距离权重计算（基于地理位置）
  3. 信誉分权重
  4. 时间偏好匹配
输出：匹配分数 + 推荐理由
```

### 6.2 需求分类模型

```
输入：用户发布的自然语言需求
处理：
  1. 文本预处理
  2. 需求类别识别（技能/厨房/时间/应急）
  3. 紧急程度判断
  4. 关键词提取
输出：分类标签 + 紧急程度 + 关键词
```

### 6.3 安全风控系统

```
输入：用户行为数据
处理：
  1. 异常行为检测（频繁发布、虚假信息）
  2. 内容审核（敏感词、违规内容）
  3. 账号风险评估
输出：风险等级 + 处理建议
```

## 7. 非功能性需求

### 7.1 性能要求
| 指标 | 目标值 |
|------|--------|
| API 响应时间 | < 200ms (P95) |
| 并发用户数 | 1000+ |
| 数据库查询 | < 50ms |
| 消息推送延迟 | < 1s |

### 7.2 安全要求
- 用户数据加密存储（AES-256）
- 敏感信息脱敏展示
- 接口鉴权（JWT）
- 防 SQL 注入、XSS 攻击
- 日志审计

### 7.3 可用性要求
- 服务可用性 > 99.9%
- 数据备份：每日全量 + 实时增量
- 灾备方案：异地多活

## 9. 风险与应对

| 风险 | 影响 | 应对策略 |
|------|------|---------|
| 用户冷启动 | 平台无活跃度 | 初期运营活动、种子用户激励 |

| 服务不稳定 | 用户流失 | 云服务 + 监控告警 + 灾备 |
